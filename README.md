# Business Cost Tracker Dashboard

A full-stack web app for logging, categorizing, and visualizing business expenses. Built as a capstone project scaffold: a REST API backend with authentication and a database, plus a dashboard frontend with charts, filtering, and budget tracking.

## What's included

- **Backend** (`backend/`): Node.js + Express API with JWT authentication, a Postgres database (works with Vercel Postgres, Render Postgres, Railway, Neon, or Supabase — any standard connection string), and endpoints for expenses, categories, and analytics.
- **Frontend** (`public/index.html`): A single-page dashboard (vanilla JS + Chart.js, no build step) with spend summaries, monthly trend charts, category breakdowns, budget-vs-actual comparison, an expense table with filters/pagination, and an add-expense form. Served automatically both locally (via Express) and on Vercel (as a static asset).
- **Synthetic dataset** (`backend/data/`): 513 generated expense records across 8 categories and ~25 vendors, spanning 12 months (July 2025–June 2026) with realistic seasonal variation (e.g., marketing spend rising in Q4, utilities peaking in summer/winter). Generated by `generate_dataset.py`.
- **Deploy-ready for both Render and Vercel** — see `DEPLOYMENT.md` and `DEPLOYMENT_VERCEL.md`.

## Requirements

- Node.js 18+ and a Postgres connection string (or use the zero-setup local fallback below).

## Setup

**Option A — no account needed, run locally with an embedded database:**

```bash
cd backend
npm install
npm run seed:pglite
npm run dev:pglite
```

**Option B — against a real Postgres database** (recommended before deploying):

```bash
cd backend
npm install
POSTGRES_URL="postgres://..." npm run seed
POSTGRES_URL="postgres://..." npm start
```

Then open **http://localhost:4000** in your browser. The frontend is served automatically by the backend.

**Demo login:** `admin@costtracker.local` / `password123`

## API overview

| Endpoint | Description |
|---|---|
| `POST /api/auth/login`, `/api/auth/register` | Authentication (returns a JWT) |
| `GET/POST/PUT/DELETE /api/expenses` | CRUD for expense entries, with filtering (`category`, `vendor`, `from`, `to`) and pagination |
| `GET/POST/PUT/DELETE /api/categories` | Manage expense categories and their monthly budgets |
| `GET /api/analytics/summary` | Total spend, spend by category, top vendors |
| `GET /api/analytics/monthly-trend` | Spend per category per month, for the trend chart |
| `GET /api/analytics/budget-status` | Budget vs. actual spend for the most recent month in the data |

## Data model

- `users` — id, name, email, password_hash, role
- `categories` — id, name, monthly_budget
- `expenses` — id, category_id (FK), vendor, amount, date, description, created_by (FK)

## Regenerating the synthetic dataset

```bash
cd backend/data
python3 generate_dataset.py
```

This overwrites `categories.csv` and `expenses.csv`. Re-run `npm run seed` afterward to reload the database.

## Notes for the capstone write-up

A few things worth including in your report/presentation:

- **Architecture**: three-tier design — SPA frontend, REST API, relational database — a standard pattern worth diagramming (client → Express routes → SQLite).
- **ER diagram**: derive it directly from the data model above (three tables, two foreign keys).
- **Why synthetic data**: real company expense data is typically confidential; the seasonal-variation logic in `generate_dataset.py` is worth explaining as a deliberate modeling choice, not random noise.
- **Auth**: passwords are hashed with bcrypt and sessions use short-lived JWTs — worth a paragraph on why (no plaintext password storage, stateless auth).
